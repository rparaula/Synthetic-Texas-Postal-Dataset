The most pressing issue is how to determine the shipping details of each package. Within our MySQL database, we need to keep track the package lifecycles of 2 different package flow types: Peer-to-Peer (P2P) and Business-to-Customer (B2C). P2P packages are when one customer sends an item or package to a post office (origin facility) to be delivered to another customer. while B2C packages are when a customer orders a product online thus that business delivers the product to a mail processing facility (origin facility) to be deliver to that customer.

The full package lifecycle for each flow type looks like this:

 - (B2C) customer orders product from business -> business sends product to mail processing facility (origin facility) -> product is packaged and sorted -> package is sent to a network facility or other mail processing facility that is near destination -> package is sent to post office that is near destination -> package is either picked up, delivered locally, or sent to Smartlocker.


 - (P2P) post office (origin facility) receives item(s) from one customer wanting to deliver to another customer -> post office packages and sorts item(s) -> package departs from post office and is sent to nearest network facility -> package is sent to mail processing facility nearest to destination -> package sent to network facility near destination -> package sent to post office that is closest to destination -> package is either picked up, delivered locally, or sent to Smartlocker.


Thus, to simulate realistic trends I plan to use the Olist E-Commerce dataset to simulate realistic package order frequency and realistic timestamps for package received, sorting, and departing at the origin facility. For realistic timestamps for middle-mile logisitics, I plan to use the Delhivery dataset for inspiration.

- For the shipping details of each package, how that is generated will depend on the type of package flow: 
For example, for B2C packages we can assume that the origin facility will be the mail processing facility that is closest to the territoryID of the business that is delivering the product to us. For the packages destination, it will simply be the territoryID of the customer or the customer's preferred facility depending on whether the package is a delivery, pickup, or smartlocker.

For P2P packages, the package has to originate from one customer and be delivered to another. The two customers involved within a P2P package exchange must be greater than a minimum distance baseline, this is to ensure that no P2P sender is so close to a P2P recipient as it wouldn't make sense for a customer to send a package via USPS to someone else if they were in driving distance. Since we're using Olist dataset relationship of order -> customer to map our P2P package -> sender, we'll also use the preferred facility ID of every customer as the origin facility (which should also always be a post office facility).

Sorted by an "Logistics" department of that post office, and departed from an "Delivery" employee of that same post office. 

Lastly, the to_facility_id must also make sense. For B2C packages, we have to check if there is a mail processing/network facility that is closer to the destinationID their their origin facility. If there is the package would go there first, if not they would go to the postal office that is closest to the destination territoryID. For P2P packages, they are checking for any network facilities or mail processing facilities that closer to the destination their the P2P package's origin post office. If there is a network/mail processing facility closer the packages goes there, if not the package goes directly to the destination.
